pki.io

Scalable and open source X.509 certificate management

Our aim: Help make network encryption the norm

pki.io is an open source project that aims to make managing and deploying X.509 certificates easy and scalable.

Protecting your internal traffic is essential, whether for compliance or simply following security best practices, and pki.io is being designed to help you easily manage certificates for internal infrastructure traffic, SSL and IPSec VPN connections, intranet and operations services, as well as implement X.509 mutual authentication.

Install the agent on your servers. Use tags to identify running services.

Create as many CAs as you need, for example one for each tagged service.

Automatically sign and distribute certificates to your servers.

pki.io is in the early design stage of development. If you're interested in helping out in any way, contact us at help@pki.io or on twitter at @pki_io.

Design Goals

Goals

  • Secure by design
  • Secure install by default
  • Scalability by design
  • Scalability through automation
  • Easy to use
  • Easy administration
  • Private data is encrypted
  • Everything is signed (authenticated)
  • Reuse existing trusted platforms (e.g. SSH or Configuration Management systems)
  • A compromise of the API service must only affect availability, not confidentiality or integrity
  • Important data is automatically backed up
  • Only selected admins (or admin processes) hold the keys to the kingdom
  • It should be possible to recover from the compromise of one or more admin systems
  • It should ideally be possible to recover from the activity of a single malicious admin user
  • A malicious system must not be able to impersonate an existing system
  • No limits to the number of CAs or certificates used
  • Publicly reviewable code
  • Built using an appropriate language
  • Developed using SDLC best practices

Design and architecture


pki.io is made up of three main components. The agent that runs on client servers; admin users or an admin service; and an API service that glues everything together. The client agent is responsible for generating and uploading CSRs as well as downloading and deploying the certs. Admins can either be users or an admin service, and they look after the CAs and sign certs etc. All connections from the agents and admins are outbound to the API service which acts as temporary and persistent data storage. The API service and admin service are run active-active for High Availability.

Like a traditional X.509 PKI, there is a root CA. This CA is expected to be valid for a very long time (e.g. 10 years) and the private keys are stored securely offline. The root CA is used to sign an intermediate CA which for pki.io is the Org CA. All the admins have access to the Org's private keys so they can act on behalf of the Org. The Org CA is then itself used to sign service CAs which are used for client certificates. We use multiple service CAs a means of separating services. Revocation dosesn't then depend on CRLs because we can simply change the service CA and regenerate and deploy all the certs automatically. Also, it makes mutual authentication simpler because we can authenticate and authorize any valid certificate from a certain service CA knowing that only relevant clients have access.

All sensitive data (such as private keys) is encrypted and all data is authenticated using signatures. For example, if an admin creates a new CA, then they would encrypt the CA private key using something like AES, then the AES key would be encrypted using the Org's public encryption key, and the two would be combined and signed by the orgs private signing key. All three bits of information would be transported together.
